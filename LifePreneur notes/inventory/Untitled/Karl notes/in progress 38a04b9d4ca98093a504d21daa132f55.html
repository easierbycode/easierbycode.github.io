Title: Add /api/product-by-url — resolve a TikTok product-page URL to a product

Repo: easierbycode/data-pimp (serves https://thirsty.store). Work on a new
feature branch, commit, push, and open a ready-for-review PR against main.

Context:
data-pimp already calls ScrapeCreators to resolve TikTok products. The inventory
app (admin.thirsty.store) lets a user paste/share a TikTok product URL and needs
this endpoint to turn that URL into a product so it can be saved as a sample.

Goal:
Add GET /api/product-by-url?url=<url-encoded product page url> that:
  1. Extracts the numeric TikTok product id from the URL. Support at least:
       https://shop.tiktok.com/view/product/<ID>?source=...
       https://www.tiktok.com/shop/pdp/<slug>/<ID>
       https://shop.tiktok.com/.../<ID>
     and short links (vt.tiktok.com / vm.tiktok.com / t.tiktok.com): follow the
     redirect SERVER-SIDE to get the final URL, then extract the id.
  2. Looks the id up via ScrapeCreators (the same call /api/product-lookup uses).
  3. Returns this shape (matches the inventory app's UpcMatch contract):
       {
         ok: boolean,
         productId: string,            // "1731435940099691085"
         name: string | null,
         price: number,                // 0 when unknown (NOT null)
         image: string | null,
         seller: string | null,
         sourceUrl: string | null,     // canonical TikTok PDP url
         source: 'tiktok',
         product: <raw ScrapeCreators response>,   // app stores this as product_json
         error?: string
       }

Requirements:
- 400 with a clear error when no id can be extracted or the url isn't a TikTok url.
- Reuse the existing ScrapeCreators lookup + field mapping so this stays
  consistent with /api/product-lookup and /api/upc-lookup.
- Add ?debug=1 to echo the raw ScrapeCreators payload (mirror the other endpoints).
- Persistence is NOT this endpoint's job: the inventory app owns the samples
  Postgres and saves the sample (qr_code = productId, product_json = product).

Verify:
  curl "https://<host>/api/product-by-url?url=https%3A%2F%2Fshop.tiktok.com%2Fview%2Fproduct%2F1731435940099691085%3Fsource%3DliveManager%26region%3Dus" | jq
  -> { ok:true, productId:"1731435940099691085", product:{...}, ... }
Title: Return a ranked `candidates` array from the UPC lookup endpoint

Repo: easierbycode/data-pimp. Feature branch, commit, push, ready-for-review PR.

Context:
GET /api/upc-lookup/{upc} resolves a UPC via a pipeline (UPCitemdb -> SerpApi
Google Shopping + Lens -> ScrapeCreators TikTok match) and today returns only the
single best `match`. In testing that auto-picked first result is frequently wrong
(3rd-party / spam SKUs reusing the UPC). The inventory app now renders a
match-picker that consumes a ranked candidates array — this task surfaces it.

Goal:
Make GET /api/upc-lookup/{upc} (and ?debug=1) return a new top-level `candidates`
array — ranked best-first — IN ADDITION to the existing fields. Don't remove or
rename anything.

Candidate shape (UpcMatch):
  {
    productId: string | null;   // TikTok product id when known
    name: string | null;
    price: number;              // 0 when unknown (NOT null)
    image: string | null;
    seller: string | null;
    sourceUrl: string | null;
    source?: string | null;     // 'tiktok' | 'google_shopping' | 'google_lens' | 'upcitemdb'
    product?: unknown;          // raw upstream record
  }
Full response: { ok, upc, upcItem?, match?, candidates?, error? }

Requirements:
1. `candidates` ranked best-first; set `match = candidates[0]` (keep `match` for
   backward compatibility — the app falls back to it).
2. Each candidate uses the exact field names/types above (price is a number, use
   0 not null when unknown). Set `source` on every candidate.
3. Cap ~10 candidates; de-dupe (same productId or normalized title+seller).
4. Preserve ?debug=1 behaviour (still echo the raw SerpApi payloads). `candidates`
   appears with or without debug.
5. Zero matches -> candidates: [] and match: null.

Note: the candidate data already exists internally (the Google Shopping results
array + any ScrapeCreators matches) — this is about collecting them into one
ranked, mapped list instead of discarding all but the top one.

Verify:
  curl "https://<host>/api/upc-lookup/631656703559" | jq
  -> candidates[] with >1 entry for a multi-result UPC; match deep-equals candidates[0].
Title: Add /api/lens-lookup — Google Lens image search returning ranked candidates

Repo: easierbycode/data-pimp. Feature branch, commit, push, ready-for-review PR.

Context:
data-pimp already runs SerpApi Google Lens as part of the UPC pipeline. The
inventory app now has a "Find by photo" flow: it hosts the photo at a public URL
(admin.thirsty.store/api/image/<id>) and needs this endpoint to match it.

Goal:
Add GET /api/lens-lookup?image=<url-encoded public image url> that runs Google
Lens (SerpApi) on the image and returns the SAME candidate shape as upc-lookup.

Response:
  interface UpcMatch {
    productId: string | null; name: string | null; price: number; // 0 if unknown
    image: string | null; seller: string | null; sourceUrl: string | null;
    source?: string | null;   // 'google_lens' | 'tiktok'
    product?: unknown;
  }
  { ok, image, match?: UpcMatch | null, candidates?: UpcMatch[], error? }

Requirements:
1. `candidates` ranked best-first; `match = candidates[0]`.
2. Map each Lens visual match to UpcMatch. Where a Lens result resolves to a
   TikTok Shop listing via ScrapeCreators, prefer/boost it and tag source
   'tiktok'; otherwise tag 'google_lens'.
3. price is a number (0 when unknown); de-dupe; cap ~10.
4. Support ?debug=1 to echo the raw Lens payload (mirror upc-lookup).
5. Validate `image` is an http(s) URL; 400 on missing/invalid.
6. Reuse the same UpcMatch mapping/ranking helpers as the upc-lookup work so the
   endpoints stay consistent.

Verify:
  curl "https://<host>/api/lens-lookup?image=https%3A%2F%2Fadmin.thirsty.store%2Fapi%2Fimage%2F<id>" | jq
  -> candidates[] of UpcMatch objects, match == candidates[0].
  Missing/garbage image -> 400, no crash.
